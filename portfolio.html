お題とタイマーで始めて、講評で次の一枚につなげる。

これは何か

Overview

お題を引き、タイマーで時間を計って紙に描き、スマホで撮って見せると、 言語モデルがコーチとして講評を返す。 講評は必ず いいところ → のびしろ → つぎの一歩 の順で、 改善点は最大2つまで。練習の記録は端末に貯まり、続けるほど自分の変化が見える。

お手本（模写の元）を渡せば「見比べ講評」、渡さなければ想像描き・落書きの講評になる。 描く体験そのものはあえてアプリ化せず、紙とえんぴつの練習に伴走する道具として設計した。

設計思想

Design Philosophy

講評の「優しさ」を仕様にした。 絵の練習が続かない最大の理由は技術ではなく挫折にある。だからコーチの人格をプロンプトで固定した—— 良い点を先に具体的な箇所で褒める・改善は「次の一枚に効くこと」だけを1〜2個・ 人格や才能ではなく絵の事実だけを語る。優しさは飾りではなく、継続のための機能として設計している。

描く体験は紙に残した。 ブラウザにキャンバスを作ることもできたが、描き心地は本物の紙に勝てない。 アプリの役割を「お題 → 時間 → 講評 → 記録」の伴走に絞り、 描く行為はいちばん気持ちいい道具（紙・えんぴつ・板タブ何でも）に任せる分業にした。

制約を設計に変えた。 お金をかけない・サーバーを持たない・個人で使う。 鍵は各自が持ち込み（BYOK）、データは端末の localStorage、通信は無料枠のみ。 姉妹アプリ（語彙ラボ・SnapTeX 等）と同じ土台なので、同一オリジン上では保存済みのキーをそのまま流用できる。

技術構成

Architecture

配信形態
単一HTMLファイル。ビルド工程・フレームワーク・依存パッケージなし。ダブルクリックでも静的ホスティングでも動く。
AI 呼び出し
ブラウザから Gemini API（vision）を直接叩く。お手本と自分の絵の2枚＋講評プロンプトを送り、SSEストリーミングで受信しながら逐次表示。
画像前処理
スマホ写真（数MB）は送信前に canvas で長辺1280px・JPEGへ縮小。無料枠の消費と通信量を1/10以下に抑える。
永続化
localStorage に練習履歴（サムネイル・メモ・講評全文）。容量あふれは古い記録から自動で間引く。
オフライン
お題辞書とタイマーはローカル実装なので、鍵もネットもない状態で練習だけは始められる。

主要な技術的判断

Key Decisions — 課題 / 判断 / 理由

「思考モード」の世代差を吸収する

課題
最新のGeminiは既定で思考モードが有効になり、講評を書き始めるまで数十秒待たされる。しかも無効化の指定方法がモデル世代ごとに違い、間違えるとAPIエラーになる。
判断
モデル名から思考設定の候補列を組み立て（3系=thinkingLevel:"minimal"／2.5系=thinkingBudget:0／2.0系=指定なし）、思考設定が原因の400なら同じモデルのまま次の候補で再試行する。実体が変わる別名（flash-latest）は全候補を順に試す。
理由
モデルは今後も入れ替わる。「正しい指定を知っている」ことに頼らず、「間違えても自動で立て直す」構造にしておく方が長持ちする。

ストリーミングで体感速度を設計する

課題
講評は文章量があり、全文を待ってから表示すると生成が速くても「遅いアプリ」に感じる。
判断
streamGenerateContent（SSE）に切り替え、受信した端から講評カードを描画。開始までの間は経過秒数を表示し、書き始めた瞬間に結果へスクロールする。
理由
体感の待ち時間は「最初の文字が出るまで」で決まる。思考モードの抑制と合わせ、待ちの数十秒を数秒に縮めた。

講評フォーマットを通信プロトコルにする

課題
LLMの出力は形式が揺れる。JSONを要求すると壊れたときに全滅し、自由文だと構造化表示ができない。
判断
3つの絵文字見出しをマーカーとしてプロンプトで固定し、画面側でマーカーを線画SVGの見出しカードに変換する。マーカーが検出できない出力は、そのまま全文表示にフォールバックする。
理由
「構造化はしたいが、壊れても読める」を両立する最小の取り決め。ストリーミング中の不完全なテキストでも段階的に描画できる。

表示バグから検証を強化する

課題
公開直後、講評カードが一度も表示されないバグが出た。CSSの display:none に対し style.display="" を代入していた——空文字はインラインスタイルの削除であり、表示にはならない。ヘッドレス検証はインライン値だけを見ていたため素通りした。
判断
表示確認を getComputedStyle（実際に画面へ出ているか）基準に改め、講評パース・履歴・画像縮小・エラー処理を実行するテストコードをヘッドレスブラウザに注入する検証を毎回通す。
理由
「編集が成功した」と「動く」は別物。バグは直すだけでなく、同じ種類のバグを二度と通さない検査に変換する。

機能マップ

Features

練習する

お題ジェネレータ（6ジャンルのローカル辞書） ジェスチャードローイング用タイマー（30秒〜5分） 終了チャイム（WebAudio・外部音源なし）

見せる・講評

お手本との見比べ講評（お手本は任意） スマホカメラでそのまま撮影 いいところ→のびしろ→つぎの一歩 講評のストリーミング表示 講評コピー

続ける

練習履歴（サムネ＋メモ＋講評全文） お題の自動記録 タップで講評を読み返す

土台

モデル自動切替（429/廃止に耐える） 思考モード自動抑制 画像の自動縮小 姉妹アプリとのAPIキー共有

制約と割り切り

Honest Limitations

承知の上で選んだトレードオフ

  • 講評の目はモデルの視覚理解に依存する。 細かなデッサンの狂いを見落とす・逆に言い過ぎることがある。だからこそ改善点を最大2つに絞り、「網羅的な採点」ではなく「次の一枚への声かけ」に役割を限定した。
  • APIキーは端末内に保存＝個人利用が前提。 不特定多数への提供にはプロキシが要る。ファイルを渡す形なら各自が自分の鍵を入れるので、無料のまま安全に配れる。
  • アプリ内では描けない。 意図的な分業だが、「その場でちょっと直して再提出」の速度は犠牲にしている。描画は将来も専用ツールに任せる方針。

この作品の位置づけ

Portfolio Axis

わたしの制作物に共通する軸は、
「学習と創作の体験を設計すること」。

音楽理論ゼロでも曲が完成する Chord Lab、語彙が自分専用に育つ語彙ラボ、 文学を理解しながら読める栞——テーマは違っても、狙いはいつも 「そのままでは続かないものを、続けたくなる体験に変える」こと。 スケッチコーチは、その軸を「絵の練習」に向けたもの。 上手さを測る道具ではなく、また描きたくなる理由を返す道具として作った。
